## Redis 序列化异常

### 🔍 问题现象

#### 前端控制台错误
```javascript
// 浏览器控制台报错
✅ 从配置构建字典完成，共 0 个  // ← 应该是 10 个
⚠️ 字典未注册：salespersons
⚠️ 字典未注册：orderStatus
⚠️ 字典未注册：documentStatus
❌ 主表格查询失败：Error: 查询失败：未找到模块 [saleorder] 对应的表名配置
```

#### 后端日志
```log
INFO  ERPConfigParser.js:64  JSON 解析成功
INFO  ERPConfigParser.js:84  数据库配置加载成功：销售订单管理
INFO  ERPConfigParser.js:85 📦 配置版本：N/A
```

### 🎯 根本原因

**Redis 缓存中存储的数据包含 FST 序列化产生的二进制前缀**，导致前端 JavaScript 无法正确解析 JSON 数据。

**技术细节**：
```bash
# Redis 实际存储的内容（错误格式）
$ redis-cli HGETALL erp_config
Key: "saleorder"
Value: "\x00\x00\x00\x00\x00\x00\x00\x00\xd7m\x00\x00\"{\\\"pageConfig\\\":{...}}"
       ↑ ↑ ↑ ↑ ↑ ↑ ↑ ↑ ↑ ↑ ↑ ↑
    这些是 FST 序列化的二进制头，导致 JSON.parse() 失败

# 前端期望的内容（正确格式）
$ redis-cli HGETALL erp_config
Key: "saleorder"
Value: "{\"pageConfig\":{\"title\":\"销售订单管理\",...}}"  # 纯 JSON 字符串
```

### 🔬 原因分析

#### 1. **Redisson 序列化器不兼容**

**问题根源**：
- Redisson 默认使用 **FST 序列化**或**JSON 序列化**
- FST 序列化会产生二进制格式的数据
- JavaScript 的 `JSON.parse()` 只能解析纯文本 JSON

**序列化对比**：

| 序列化方式 | 格式 | 可读性 | 前端解析 | 跨语言 | 性能 |
|-----------|------|--------|---------|-------|------|
| **FST** | 二进制 | ❌ 不可读 | ❌ 无法解析 | ❌ 仅 Java | ✅ 快 |
| **JSON** | 文本 | ✅ 人类可读 | ✅ JSON.parse() | ✅ 所有语言 | ⚠️ 中等 |

**当前配置（错误）**：
```java
// RedisConfig.java - 可能使用了 FST 序列化
config.setCodec(new FstCodec());  // ❌ 错误
```

#### 2. **前后端共享缓存的架构冲突**

**架构问题**：
```
Java 后端 --[FST 序列化]--> Redis --[二进制数据]--> JavaScript 前端
                                      ↓
                              JSON.parse() 失败
```

**正确的架构应该是**：
```
Java 后端 --[JSON 序列化]--> Redis --[纯 JSON]--> JavaScript 前端
                                     ↓
                             JSON.parse() 成功 ✅
```

### 🛠️ 修复方案

#### 方案 1：修改 Redis 序列化配置（推荐）⭐

**修改文件**：`ruoyi-common-redis/src/main/java/.../RedisConfig.java`

**修改前**：
```java

```

**修改后**：
```java
@Bean
public RedissonAutoConfigurationCustomizer redissonCustomizer() {
    return config -> {
        ObjectMapper om = new ObjectMapper();
        // ... 配置 ObjectMapper
        
        TypedJsonJacksonCodec jsonCodec = new TypedJsonJacksonCodec(Object.class, om);
        
        // ✅ 正确的组合方式：key 和 value 都使用纯 JSON 格式
        CompositeCodec codec = new CompositeCodec(
            StringCodec.INSTANCE,  // Key: 纯字符串
            jsonCodec,             // Hash Key: JSON
            jsonCodec              // Hash Value: JSON
        );
        
        config.setCodec(codec);
        // ... 其他配置
    };
}
```

**关键改动说明**：
```java

TypedJsonJacksonCodec jsonCodec = new TypedJsonJacksonCodec(Object.class, om);

// ✅ 组合序列化确保：
// - Key 使用 String（纯文本）
// - Hash Key 使用 JSON（纯文本）
// - Hash Value 使用 JSON（纯文本）
CompositeCodec codec = new CompositeCodec(
    StringCodec.INSTANCE,  // Key: 纯字符串
    jsonCodec,             // Hash Key: JSON
    jsonCodec              // Hash Value: JSON
);
```

```java
@RestController
@RequestMapping("/erp/cache")
public class CacheHealthController {
    
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    
    @GetMapping("/health")
    public R<?> checkCacheHealth() {
        try {
            // 测试写入
            String testKey = "health_check:" + System.currentTimeMillis();
            String testValue = "{\"status\":\"ok\"}";
            
            redisTemplate.opsForValue().set(testKey, testValue);
            
            // 测试读取
            String value = redisTemplate.opsForValue().get(testKey);
            
            // 验证是否为纯 JSON
            if ("{\"status\":\"ok\"}".equals(value)) {
                redisTemplate.delete(testKey);
                return R.ok("缓存正常，使用 JSON 序列化");
            } else {
                return R.fail("缓存格式异常：" + value);
            }
        } catch (Exception e) {
            return R.fail("缓存检查失败：" + e.getMessage());
        }
    }
    
    @GetMapping("/inspect/{key}")
    public R<?> inspectCache(@PathVariable String key) {
        try {
            String value = (String) redisTemplate.opsForValue().get(key);
            
            if (value == null) {
                return R.ok(Map.of("exists", false));
            }
            
            // 检查是否包含二进制字符
            boolean hasBinary = !value.matches("\\p{Print}*");
            
            return R.ok(Map.of(
                "exists", true,
                "hasBinary", hasBinary,
                "length", value.length(),
                "preview", value.substring(0, Math.min(100, value.length()))
            ));
        } catch (Exception e) {
            return R.fail("检查失败：" + e.getMessage());
        }
    }
}
```

## 表名映射配置缺失

### 🔍 问题现象

#### 后端日志
```log
ERROR c.r.s.c.erp.ErpEngineController - 动态查询失败
com.ruoyi.common.core.exception.ServiceException: 
未找到模块 [saleorder] 对应的表名配置
    at SuperDataPermissionServiceImpl.getTableNameByModuleCode(SuperDataPermissionServiceImpl.java:161)
```

#### 前端错误
```javascript
❌ 主表格查询失败：Error: 查询失败：未找到模块 [saleorder] 对应的表名配置
```

### 🎯 根本原因

**旧代码硬编码了 moduleCode 与 tableName 的映射关系**，但新架构要求从 JSON 配置直接获取 tableName。

**错误的代码逻辑**：
```java
// ❌ 旧代码：硬编码映射
private static final Map<String, String> MODULE_TABLE_MAPPING = new HashMap<>();

static {
    MODULE_TABLE_MAPPING.put("saleOrder", "t_sale_order");  // ❌ 硬编码
    MODULE_TABLE_MAPPING.put("saleOrderEntry", "t_sale_order_entry");
}

@Override
public String getTableNameByModuleCode(String moduleCode) {
    String tableName = MODULE_TABLE_MAPPING.get(moduleCode);
    if (tableName == null) {
        throw new ServiceException("未找到模块 [" + moduleCode + "] 对应的表名配置");
    }
    return tableName;
}
```

**问题**：
1. ❌ 每新增一个模块就要修改 Java 代码
2. ❌ 需要重新编译、重启服务
3. ❌ moduleCode 大小写不一致会导致查找失败
4. ❌ 违反了低代码的配置驱动原则

### 🛠️ 修复方案

#### 方案 1：使用 JSON 配置传递 tableName（推荐）⭐

**新的架构**：
```javascript
// JSON 配置文件（数据库存储）
{
  "pageConfig": {
    "tableName": "t_sale_order"  // ✅ 直接从配置读取
  },
  "queryConfig": {
    "conditions": [...],
    "orderBy": [...]
  }
}
```

**后端实现**：
```java
// ✅ 新方法：直接使用 tableName
@Override
public Page<Map<String, Object>> selectPageByTableName(
        String tableName,  // ✅ 直接从 JSON 配置传递
        PageQuery pageQuery,
        QueryWrapper<Object> queryWrapper) {
    
    try {
        // ✅ 不需要查找映射，直接使用 tableName
        String sql = buildSelectSql(tableName, queryWrapper);
        String countSql = buildCountSql(tableName, queryWrapper);
        
        // ... 执行查询
        log.info("动态查询成功，tableName: {}, total: {}", tableName, total);
        
        return page;
    } catch (Exception e) {
        log.error("动态查询失败，tableName: {}", tableName, e);
        throw new ServiceException("查询失败：" + e.getMessage());
    }
}

// ⚠️ 废弃旧方法（抛出异常强制迁移）
@Override
@Deprecated
public String getTableNameByModuleCode(String moduleCode) {
    throw new ServiceException("表名必须从 JSON 配置获取，禁止使用模块编码映射");
}
```

**前端调用**：
```javascript
import ERPConfigParser from '@/views/erp/utils/ERPConfigParser'

const getList = async () => {
  // ✅ 从 JSON 配置读取 tableName
  const config = await ERPConfigParser.loadFromDatabase(moduleCode)
  const tableName = config.pageConfig.tableName
  
  const response = await request({
    url: '/erp/engine/query/execute',
    method: 'post',
    data: {
      moduleCode: moduleCode,
      tableName: tableName,  // ✅ 传递 tableName
      queryConfig: config.queryConfig
    }
  })
}
```

#### 方案 2：确保 JSON 配置包含 tableName

**SQL 验证**：
```sql
-- 检查所有模块的配置

    module_code,
    config_name,
    JSON_EXTRACT(config_content, '$.pageConfig.tableName') as table_name

-- 检查结果

------------|------------------|------------------
```

```sql
UPDATE erp_page_config 
SET config_content = JSON_SET(

    '$.pageConfig.tableName',

        WHEN module_code = 'saleorder' THEN 't_sale_order'
        WHEN module_code = 'purchaseorder' THEN 't_purchase_order'
        WHEN module_code = 'deliveryorder' THEN 't_delivery_order'
        -- 根据实际情况添加更多映射
    END
)
WHERE JSON_EXTRACT(config_content, '$.pageConfig.tableName') IS NULL;
```

## moduleCode 大小写不一致

### 🔍 问题现象

#### 错误日志
```log
ERROR: 未找到模块 [saleorder] 对应的表名配置
```

#### 配置对比
```json
// ❌ 错误：数据库使用小写，配置使用驼峰
{
  "pageConfig": {
    "moduleCode": "saleOrder"  // 驼峰
  }
}

// ✅ 正确：统一为小写
{
  "pageConfig": {
    "moduleCode": "saleorder"  // 全小写
  }
}
```

### 🎯 根本原因

**Java 的 Map 查找区分大小写**，导致 `saleorder` ≠ `saleOrder`。

### 🛠️ 修复方案

#### 方案 1：统一为全小写（推荐）

**批量修改数据库**：
```sql
-- 将所有 module_code 转换为小写
UPDATE erp_page_config 
SET module_code = LOWER(module_code)
WHERE module_code != LOWER(module_code);

-- 同时更新 config_content 中的 moduleCode
UPDATE erp_page_config 
SET config_content = JSON_SET(
    config_content, 
    '$.pageConfig.moduleCode', 
    LOWER(JSON_EXTRACT(config_content, '$.pageConfig.moduleCode'))
);
```

**前端配置检查**：
```javascript
// 验证 moduleCode 格式
if (!/^[a-z]+$/.test(config.pageConfig.moduleCode)) {
    console.error('moduleCode 必须为全小写字母:', config.pageConfig.moduleCode);
    // 自动转换为小写
    config.pageConfig.moduleCode = config.pageConfig.moduleCode.toLowerCase();
}
```

#### 方案 2：忽略大小写查找

**后端修改**：
```java
// ❌ 不推荐：忽略大小写会导致性能下降
@Override
public String getTableNameByModuleCode(String moduleCode) {
    // 查找时忽略大小写
    for (Map.Entry<String, String> entry : MODULE_TABLE_MAPPING.entrySet()) {
        if (entry.getKey().equalsIgnoreCase(moduleCode)) {
            return entry.getValue();
        }
    }
    throw new ServiceException("未找到模块 [" + moduleCode + "] 对应的表名配置");
}
```

## 字典未注册

### 🔍 问题现象

#### 前端控制台警告
```javascript
DictionaryBuilder.js:227 ⚠️ 字典未注册：salespersons
get @ DictionaryBuilder.js:227
saleorder.vue:935 ⚠️ 字典未注册：salespersons
getDictOptions @ saleorder.vue:935

DictionaryBuilder.js:331 ✅ 从配置构建字典完成，共 0 个  // ← 应该是 10 个
```

### 🎯 根本原因

**Redis 缓存数据格式损坏**，导致前端无法解析 JSON，字典构建器无法从配置中注册字典。

**错误链路**：
```
Redis 缓存损坏 → JSON 解析失败 → 配置内容为空 → 字典未注册 → 下拉框无数据
```

### 🛠️ 修复方案

参考 **[Redis 序列化异常](#redis-序列化异常)** 章节的完整修复流程。

**快速修复步骤**：

1. **清理 Redis 缓存**
```powershell
redis-cli -h localhost -p 6379 -a difyai123456 DEL "erp_config"
```

2. **修改 RedisConfig.java**（见上文）

3. **重新编译并重启**
```powershell
mvn clean package -DskipTests -pl ruoyi-admin-wms -am
.\start-services.ps1
```

4. **刷新浏览器**
```
Ctrl + F5
```

## 📝 总结

### 常见问题速查表

| 问题 | 根本原因 | 修复方法 | 耗时 |
|------|---------|---------|------|
| **Redis 序列化异常** | FST 二进制格式 | 改用 JSON 序列化器 | 10 分钟 |
| **表名映射缺失** | 硬编码映射 | 使用 JSON 配置传递 | 15 分钟 |
| **moduleCode 不一致** | 大小写敏感 | 统一为全小写 | 5 分钟 |
| **字典未注册** | 缓存损坏 | 清理缓存 + 重编译 | 10 分钟 |

### 最佳实践

1. **配置驱动优先**
   - ✅ 能用配置的，绝不硬编码
   - ✅ 能动态读取的，绝不静态绑定

2. **JSON 序列化统一**
   - ✅ Redis 缓存统一使用 JSON 格式
   - ✅ 避免 FST 等二进制序列化
