## 🔍 风险点解决情况

### 原方案的 3 个高风险点

#### 🔴 风险 1: 配置复杂度高，难以管理

**问题表现**：
- JSON 文件散落在各个目录
- 无法快速查找某个配置
- 缺少统一管理界面

**解决方案**：
 **统一公共配置表**
- 所有配置存储在 `erp_page_config` 表
- 通过 `module_code` 和 `config_type` 分类
- 提供统一的管理界面

---

#### 🔴 风险 2: 下推数据一致性问题

**问题表现**：
- 并发下推可能导致数量错误
- 源单和目标单数据不一致
- 缺少事务保证

**解决方案**：
 **专用下推关系表 + 乐观锁**
- 新建 `erp_push_relation` 表专门管理下推规则
- 使用 `version` 字段实现乐观锁
- `@Transactional` 保证事务一致性

```java
@Transactional(rollbackFor = Exception.class)
public void executePush(PushRequest request) {
    // 1. 加载源单（带版本号）
    SourceBill source = loadSourceBill(request.getSourceId());
    
    // 2. 执行映射和转换
    TargetBill target = transformData(source, request.getMappingRules());
    
    // 3. 保存目标单
    targetBillMapper.insert(target);
    
    // 4. 更新源单状态（乐观锁检查）
    int updated = sourceBillMapper.updatePushedQty(
        source.getId(), 
        request.getPushedQty(),
        source.getVersion()  // 乐观锁关键
    );
    
    if (updated == 0) {
        throw new ConcurrentModificationException("更新失败，可能被其他事务修改");
    }
    // 失败时自动回滚
}
```

#### 🔴 风险 3: 权限控制弱

**问题表现**：
- 前端配置可被绕过
- 缺少后端校验
- 无法细粒度控制

**解决方案**：
 **RBAC 权限 + 后端验证**
- Controller 层添加 `@SaCheckPermission` 注解
- Service 层验证 JSON Schema
- 前端按钮级别权限控制

```java
// Controller 权限控制
@SaCheckPermission("erp:config:edit")
@PostMapping("/save")
public R<Void> saveConfig(@RequestBody SaveConfigRequest request) {
    // Service 层验证
    ValidationResult result = ConfigValidator.validate(config);
    if (!result.isValid()) {
        return R.fail("配置验证失败：" + result.getErrors());
    }
    configService.saveConfig(config, request.getChangeReason());
    return R.ok();
}

// 前端权限控制
<el-button 
  icon="Edit" 
  @click="handleEdit"
  v-hasPermi="['erp:config:edit']"
>
  编辑配置
</el-button>
```

##  常见问题解答

### Q1: 是否必须使用 Redis？

**答**: 不是必须的。简化版方案不使用 Redis 也能正常工作。Redis 是性能优化选项，建议在并发量高或性能瓶颈时再添加。

---

### Q2: 配置历史表是否必需？

**答**: 不是必需的。可以先实施主表 `erp_page_config`，历史表作为可选扩展。但强烈建议实施，因为：
-  自动记录所有变更
-  支持一键回滚
-  审计追踪

---

### Q3: 如何处理大 JSON 配置（超过 1MB）？

**答**: 三种方案：
1. **拆分配置**：按功能拆分成多个小配置（推荐）
2. **压缩存储**：使用 GZIP 压缩后存储
3. **外部存储**：存储到 OSS，数据库只存 URL

---

### Q4: 配置修改后多久生效？

**答**: 
- **不使用缓存**：立即生效
- **使用 Redis 缓存**：清除缓存后立即生效（默认 1 小时 TTL）
- **使用本地缓存**：最多 10 分钟延迟（可手动刷新）

---

### Q5: 如何保证配置的安全性？

**答**: 四层防护：
1. **权限控制**：`@SaCheckPermission` 注解
2. **Schema 验证**：JSON Schema 验证配置格式
3. **事务保护**：`@Transactional` 保证数据一致性
4. **审计日志**：所有变更自动记录到历史表
