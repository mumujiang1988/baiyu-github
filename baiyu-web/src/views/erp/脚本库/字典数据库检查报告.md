## 五、影响分析

### 5.1 对销售订单的影响

**使用的字典：**
1. ✅ order_status - sys_dict_data有3条数据
2. ✅ f_document_status - sys_dict_data有5条数据
3. ❌ currency - sys_dict_data缺失，导致结算币别、本位币选择失败
4. ❌ payment_clause - sys_dict_data缺失，导致收款条件选择失败
5. ❌ trade_way - sys_dict_data缺失，导致贸易方式选择失败

### 5.2 对后端API的影响

**后端API `/erp/engine/dict/all` 返回结果：**
- 只返回 sys_dict_data 表中存在的字典数据
- 不会返回 bymaterial_dictionary 表中的数据
- 导致前端无法获取21个字典类型的数据

### 5.3 对前端字典引擎的影响

**DictionaryBuilder.js 的工作流程：**
1. 从后端API获取字典数据
2. 根据 dict_config 配置注册字典
3. 渲染页面时使用字典数据

**问题：**
- 由于 sys_dict_data 缺失数据，后端API无法返回
- 前端字典引擎无法注册这些字典
- 页面渲染时报错：`[字典引擎] ❌ 字典未注册`

## 六、解决方案

### 6.1 方案一：从 bymaterial_dictionary 同步数据到 sys_dict_data（推荐）

**优点：**
- 数据完整，bymaterial_dictionary 已有所有业务字典数据
- 统一数据源，便于管理
- 符合系统架构设计

**步骤：**
1. 编写数据同步脚本
2. 将 bymaterial_dictionary 的数据转换并插入到 sys_dict_data
3. 注意字段映射：code → dict_value, name → dict_label, category → dict_type

### 6.2 方案二：手动补充缺失的字典数据

**优点：**
- 灵活控制数据内容
- 可以根据业务需求调整

**缺点：**
- 工作量大，需要补充21个字典类型的数据
- 容易遗漏或出错

### 6.3 方案三：修改后端API，同时查询两个表

**优点：**
- 不需要迁移数据
- 保持现有数据结构

**缺点：**
- 增加API复杂度
- 数据源不统一，维护困难
- 不符合系统设计规范

## 七、推荐执行方案

### 7.1 立即执行：创建数据同步脚本

```sql
-- 从 bymaterial_dictionary 同步数据到 sys_dict_data
INSERT INTO sys_dict_data (dict_code, dict_sort, dict_label, dict_value, dict_type, status, create_by, create_time)
SELECT
  CONCAT(203456789, LPAD(id, 10, '0')) as dict_code,
  ROW_NUMBER() OVER (PARTITION BY category ORDER BY id) as dict_sort,
  name as dict_label,
  code as dict_value,
  LOWER(category) as dict_type,  -- 统一转换为小写
  '1' as status,
  'admin' as create_by,
  NOW() as create_time
FROM bymaterial_dictionary
WHERE LOWER(category) NOT IN (
  SELECT DISTINCT dict_type FROM sys_dict_data
)
ON DUPLICATE KEY UPDATE
  dict_label = VALUES(dict_label),
  dict_value = VALUES(dict_value);
```

### 7.2 验证同步结果

```sql
-- 验证同步后的数据量
SELECT dict_type, COUNT(*) as count
FROM sys_dict_data
GROUP BY dict_type
ORDER BY dict_type;

-- 对比两个表的数据量
SELECT
  'sys_dict_data' as table_name,
  dict_type,
  COUNT(*) as count
FROM sys_dict_data
GROUP BY dict_type
UNION ALL
SELECT
  'bymaterial_dictionary' as table_name,
  LOWER(category) as dict_type,
  COUNT(*) as count
FROM bymaterial_dictionary
GROUP BY LOWER(category)
ORDER BY dict_type;
```

## 八、总结

### 8.1 核心问题
- **sys_dict_data 表缺失21个字典类型的数据**
- **bymaterial_dictionary 表有完整数据但未被使用**
- **后端API只查询 sys_dict_data，导致前端无法获取字典数据**

### 8.2 根本原因
- 数据迁移不完整，bymaterial_dictionary 的数据未同步到 sys_dict_data
- 系统设计使用 sys_dict_data 作为标准字典表，但数据未完全迁移

### 8.3 解决方案
- **推荐方案**：从 bymaterial_dictionary 同步数据到 sys_dict_data
- **执行步骤**：
  1. 执行数据同步脚本
  2. 验证同步结果
  3. 测试后端API
  4. 测试前端页面

### 8.4 预期效果
- sys_dict_data 表包含完整的字典数据（48个字典类型）
- 后端API返回完整的字典数据
- 前端字典引擎正常工作
- 销售订单页面不再报错
