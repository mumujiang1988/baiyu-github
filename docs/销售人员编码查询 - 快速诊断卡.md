### **字段映射关系**

| 位置 | 字段名 | 值类型 | 示例值 |
|------|--------|--------|--------|
| **数据库** | `sys_employee.salesman_id` | VARCHAR | "100949" |

**关键查询**:
```sql
-- 确认 100949 和 100055 的归属
SELECT 
  u.user_id,
  u.nick_name,
  e.salesman_id,
  d.dept_name
FROM sys_user u
LEFT JOIN sys_employee e ON u.staff_id = e.fid
WHERE e.salesman_id IN ('100949', '100055')
   OR u.user_id IN (100949, 100055);
```

## 💡 **可能的原因**

### **原因 1: 字段绑定错误**
```vue
<!-- ❌ 错误：使用了 value (user_id) -->
<el-select v-model="formData.FSalerId">
  <el-option
    :key="opt.value"
    :label="opt.label"
    :value="opt.value"  <!-- user_id = 1234567890 -->
  />
</el-select>

<!-- ✅ 正确：使用 fseller (salesman_id) -->
<el-select v-model="formData.fseller">
  <el-option
    :key="opt.fseller"
    :label="opt.label"
    :value="opt.fseller"  <!-- salesman_id = "100949" -->
  />
</el-select>
```

---

### **原因 2: 缓存问题**
```javascript
// 清除 localStorage 中的旧数据
localStorage.removeItem('salespersons_dict')
location.reload()
```

---

### **原因 3: 数据转换逻辑错误**
```javascript
// ❌ 可能进行了错误的转换
const salerId = salespersons.find(s => s.value === input)?.fseller
// 如果 input 是 100055，会找到错误的销售员

// ✅ 应该直接使用
const salerId = input  // 直接传入 salesman_id
```

### **方案 A: 修改前端绑定（推荐）**

**修改文件**: 使用销售人员选择器的所有页面

```diff
- <el-select v-model="formData.FSalerId">
+ <el-select v-model="formData.fseller">
    <el-option
      v-for="opt in dictionaryManager.getDict('salespersons')"
      :key="opt.fseller"
      :label="opt.label"
-     :value="opt.value"
+     :value="opt.fseller"
    />
  </el-select>
```

**优点**:
- ✅ 影响范围小
- ✅ 符合后端驱动架构
- ✅ 便于调试和维护

**缺点**:
- ⚠️ 需要修改多个页面

---

### **方案 B: 修改后端返回**

**修改文件**: `ErpDictionaryMapper.java`

```diff
@Select("SELECT " +
- "  u.user_id AS value, " +
+ "  e.salesman_id AS value, " +      // ← 改为返回 salesman_id
+ "  e.salesman_id AS FSalerId, " +   // ← 保留兼容性
"  e.salesman_id AS fseller, " +
"...")
List<Map<String, Object>> selectSalespersonsDict();
```

**优点**:
- ✅ 前端改动小
- ✅ 统一使用业务员编码

**缺点**:
- ⚠️ 可能影响其他使用 `value` 的地方
- ⚠️ 失去 user_id 的引用

## 🎯 **最佳实践**

### **字段命名规范**

```javascript
// ✅ 推荐：前后端统一
{
  fseller: "100949",        // 销售员编码（来自 database）
  fseller_name: "张三",     // 销售员姓名
  fseller_dept: "销售部"    // 销售员部门
}

// ❌ 不推荐：混用风格
{
  FSalerId: "100949",       // 金蝶风格
  salerName: "张三",        // 驼峰风格
  seller_dept: "销售部"     // 下划线风格
}
```

### **字典值设计原则**

1. **唯一性**: value 能唯一标识选项
2. **稳定性**: 使用业务编码而非自增 ID
3. **可读性**: 便于调试和日志分析
4. **一致性**: 与数据库字段保持一致
