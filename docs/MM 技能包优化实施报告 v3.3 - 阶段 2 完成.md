### 二、新增核心内容

#### 1. PDCA 持续改进循环

**核心流程图**:
```
任务执行 → 自动检查 → 经验总结 → 更新记忆/文档 → 验证效果 → 下次迭代
     ↓
  Plan-Do-Check-Act 闭环
```

**关键特性**:
- 🔄 自动化触发（每次任务完成后）
- 📊 量化评估（1-5 分评分）
- 🎯 分类处理（成功/失败不同路径）
- 📈 持续迭代（定期回顾机制）

---

#### 2. 阶段 1: 任务完成后自动检查（Check）

**检查项 1: 任务成功度评估**

| 评分 | 标准 | 行动 |
|------|------|------|
| 5 分 | 完美完成，超出预期 | 进入成功经验总结 |
| 4 分 | 良好完成，符合预期 | 进入成功经验总结 |
| 3 分 | 基本完成，有小瑕疵 | 进入异常分析流程 |
| 2 分 | 部分完成，有明显问题 | 进入异常分析流程 |
| 1 分 | 未完成，遇到阻塞 | 进入异常分析流程 |

**检查项 2: 5 类必查异常**

| 异常类型 | 典型场景 | 占比 |
|---------|----------|------|
| 🔧 工具调用失败 | search_replace 匹配、run_in_terminal 执行 | ~30% |
| 💻 命令执行错误 | SQL 语法、Maven 编译、npm 构建 | ~25% |
| 📝 配置解析错误 | JSON 格式、YAML 缩进、字段缺失 | ~20% |
| 🌐 接口调用失败 | API 路径、参数、认证授权 | ~15% |
| ⚙️ 编译/运行错误 | Java/Vue编译、运行时异常 | ~10% |

**检查项 3: 3 维度优化机会**

1. 💡 **效率提升点**
   - 重复性操作自动化
   - 更简洁的实现方式
   - 工具/脚本提效

2. 🎯 **质量提升点**
   - 代码清晰度和可维护性
   - 性能问题识别
   - 最佳实践符合度

3. 📚 **文档提升点**
   - 补充说明
   - 消除误解
   - 添加示例

#### 3. 阶段 2: 经验沉淀（Act）

**步骤 1: 5W1H 记录法**

| 要素 | 内容 | 作用 |
|------|------|------|
| **What** | 发生了什么？ | 描述异常现象 |
| **When** | 何时发生？ | 定位时间点 |
| **Where** | 在哪里发生？ | 定位文件/模块 |
| **Who** | 谁遇到的？ | 评估影响范围 |
| **Why** | 初步原因？ | 直接原因 + 触发条件 |
| **How** | 如何解决的？ | 临时方案 + 最终方案 |

**步骤 2: 5 Why 深度根因分析**

```
第 1 问 → 第 1 层原因
   ↓
第 2 问 → 第 2 层原因
   ↓
第 3 问 → 第 3 层原因
   ↓
第 4 问 → 第 4 层原因
   ↓
第 5 问 → 根本原因 ✅
```

**根本原因分类**（6 类）:
- [ ] 知识缺失
- [ ] 技能不足
- [ ] 流程缺陷
- [ ] 工具限制
- [ ] 文档不全
- [ ] 配置错误

**步骤 3: STAR 模型经验提炼**

| 要素 | 内容 | 字数要求 |
|------|------|----------|
| **Situation** | 情境描述（业务场景 + 技术背景） | ~100 字 |
| **Task** | 任务目标（原始目标 + 期望结果） | ~50 字 |
| **Action** | 采取行动（错误尝试 + 正确做法 + 关键步骤） | ~200 字 |
| **Result** | 结果验证（验证方法 + 效果对比 + 复用价值） | ~150 字 |

#### 4. 阶段 3: 知识更新（Update）

**记忆系统更新标准格式**

```json
{
  "action": "create",
  "category": "common_pitfalls_experience",
  "title": "简洁描述陷阱的标题（不超过 20 字）",
  "keywords": "关键词 1，关键词 2，关键词 3，关键词 4",
  "content": "详细的经验描述（包含问题、原因、解决方案、预防措施）"
}
```

**三类经验分类**

| 类别 | 适用场景 | 示例 |
|------|----------|------|
| **common_pitfalls_experience** | 常见陷阱、易错点、坑 | search_replace 匹配失败的预防 |
| **learned_skill_experience** | 学到的新技能、操作步骤 | MySQL 查询技能的三种连接模式 |
| **expert_experience** | 专家级领域知识、最佳实践 | ERP 配置驱动的架构设计原则 |

**优先级矩阵**（P0-P4）

| 影响范围 \\ 紧急程度 | 高（阻塞性） | 中（频繁） | 低（偶发） |
|-------------------|-------------|-----------|-----------|
| **大（所有人）** | P0 - 立即更新 | P1 - 本周更新 | P2 - 本月更新 |
| **中（部分人）** | P1 - 本周更新 | P2 - 本月更新 | P3 - 季度更新 |
| **小（个别人）** | P2 - 本月更新 | P3 - 季度更新 | P4 - 年度更新 |

**三个更新位置**:

1. **主技能文档**（SKILL.md）
   - 相关章节注意事项
   - 快速参考常见问题
   - 示例代码注释

2. **核心技能文档**（core/SKILL.md）
   - 工作流程检查点
   - 最佳实践新条目
   - 故障排查案例

3. **子技能文档**（各子技能/SKILL.md）
   - 技能概述适用场景
   - 使用指南操作步骤
   - 常见问题案例

#### 5. 阶段 4: 效果验证（Verify）

**验证方法 1: 测试用例验证**

```json
{
  "test_case_id": "TC-001",
  "scenario": "场景描述（如：查看表结构）",
  "prompt": "用户提问示例",
  "expected_output": "期望的输出结果",
  "actual_output": "实际的输出结果",
  "pass_criteria": "通过标准（是/否）",
  "notes": "备注（任何观察到的现象）"
}
```

**执行流程**:
1. 使用相同 prompt 再次执行
2. 对比实际输出与期望输出
3. 记录测试结果（通过/失败）
4. 如果失败，重新进入改进循环

**验证方法 2: 用户反馈验证**

**收集反馈**（4 个问题）:
1. 新方法是否解决了原有问题？（是/否）
2. 新方法是否易于理解和应用？（1-5 分）
3. 是否还有其他相关问题？（开放回答）
4. 有什么进一步的建议？（开放回答）

**分析反馈**:
- 满意度 ≥ 4 分 → 验证通过，可以推广
- 满意度 ≤ 3 分 → 重新进入改进循环

**验证方法 3: 数据指标验证**

| 指标 | 改进前 | 改进后 | 目标 |
|------|--------|--------|------|
| 任务完成率 | X% | Y% | ≥95% |
| 平均耗时 | X 分钟 | Y 分钟 | -20% |
| 错误发生率 | X% | Y% | -50% |
| 用户满意度 | X 分 | Y 分 | ≥4 分 |
