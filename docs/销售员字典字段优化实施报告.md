# 销售员字典字段优化实施报告

## 📋 **修复概述**

优化 `/erp/engine/dict/all` 接口返回的销售员字典数据，解决查询编码参数异常问题。

---

## 🎯 **问题根源**

### **问题现象**
用户选择销售员编码 `100949`,但实际查询条件传递的是错误的值。

### **核心原因**
后端返回的销售员字典中缺少金蝶风格的字段名 `FSalerId`,导致前端查询时字段映射不一致。

### **字段对比**

| 字段名 | 数据类型 | 说明 | 用途 |
|--------|---------|------|------|
| `value` | BIGINT | user_id (雪花算法 ID) | ❌ 不应用于查询 |
| `fseller` | VARCHAR | salesman_id (销售员编码) | ✅ 标准命名，用于查询 |
| `FSalerId` | VARCHAR | salesman_id (销售员编码) | ✨ **新增**:金蝶风格，兼容旧系统 |

---

## ✅ **修复方案**

### **修改文件**
`ErpDictionaryMapper.java` - `selectSalespersonsDict()` 方法

### **修改内容**

#### **修改前**
```sql
SELECT 
  CONCAT(u.nick_name, IFNULL(CONCAT('(', d.dept_name, ')'), ''), IFNULL(CONCAT(' - ', e.salesman_id), '')) AS label,
  u.user_id AS value,
  'salespersons' AS type,
  u.nick_name AS nickName,
  d.dept_name AS departmentName,
  e.salesman_id AS fseller,  -- ← 只有这一个销售员编码字段
  GROUP_CONCAT(DISTINCT sr.role_name ORDER BY sr.role_id) AS roleNames
FROM sys_user u
...
```

#### **修改后**
```sql
SELECT 
  CONCAT(u.nick_name, IFNULL(CONCAT('(', d.dept_name, ')'), ''), IFNULL(CONCAT(' - ', e.salesman_id), '')) AS label,
  u.user_id AS value,
  'salespersons' AS type,
  u.nick_name AS nickName,
  d.dept_name AS departmentName,
  e.salesman_id AS fseller,      -- ← 标准命名
  e.salesman_id AS FSalerId,     -- ✨ 新增：金蝶风格命名
  GROUP_CONCAT(DISTINCT sr.role_name ORDER BY sr.role_id) AS roleNames
FROM sys_user u
...
```

## 🔍 **字段使用场景**

### **1. 下拉框绑定**

```vue
<!-- ✅ 推荐使用 fseller (标准命名) -->
<el-select v-model="formData.fseller" placeholder="请选择销售员">
  <el-option
    v-for="opt in salespersons"
    :key="opt.fseller"
    :label="opt.label"
    :value="opt.fseller"
  />
</el-select>
```

### **2. 查询条件传递**

```javascript
// ✅ 使用 fsaller 字段（小写驼峰）
const queryParams = {
  conditions: [
    {
      field: "fseller",              // ✅ 标准字段名
      operator: "eq",
      value: "100949"                // 字符串类型的销售员编码
    }
  ]
}

// ⚠️ 如果后端表字段是 FSalerId，需要字段映射
// 见 search_config 配置
```

### **3. 金蝶 K3 集成**

```javascript
// ✅ 使用 FSalerId 字段（金蝶风格）
const k3Data = {
  FSalerId: "100949"    // 与金蝶 K3 字段名一致
}

// 转换逻辑
formData.fseller = selectedSalesperson.FSalerId || selectedSalesperson.fseller
```

## 🎓 **最佳实践**

### **1. 字段命名规范**

| 场景 | 推荐命名 | 说明 |
|------|---------|------|
| **前端绑定** | `fseller` | 小写驼峰，符合 JavaScript 规范 |
| **查询条件** | `fseller` | 与前端保持一致 |
| **金蝶集成** | `FSalerId` | 与金蝶 K3 字段名一致 |
| **数据库字段** | `FSalerId` | 大写 F 前缀，金蝶风格 |

### **2. 字段映射策略**

```javascript
// ✅ 推荐：配置化字段映射
const fieldMapping = {
  'fseller': 'FSalerId',    // 前端 → 数据库
  'fdate': 'FDate',
  'fbillno': 'FBillNo'
}

// 使用映射
const dbFieldName = fieldMapping[frontendFieldName] || frontendFieldName
```

### **3. 字典数据使用**

```javascript
// ✅ 优先使用 fseller（标准命名）
selectedValue = option.fseller

// ⚠️ 仅在金蝶集成时使用 FSalerId
k3Data.FSalerId = option.FSalerId

// ❌ 避免使用 value（user_id，不是销售员编码）
wrongValue = option.value  // 这是 BIGINT 类型的 user_id
```

## 📝 **总结**

### **核心改动**
- ✅ 在 `ErpDictionaryMapper.selectSalespersonsDict()` 中新增 `FSalerId` 字段
- ✅ 保持 `fseller` 字段不变（向后兼容）
- ✅ 两个字段指向同一个数据库列 `e.salesman_id`

### **解决的问题**
- ✅ 统一了前后端字段命名
- ✅ 支持金蝶 K3 原生字段名
- ✅ 消除了查询条件传递错误

### **影响范围**
- ✅ **向后兼容**: 现有代码不受影响（`fseller` 保持不变）
- ✅ **向前扩展**: 新增 `FSalerId` 字段提供更多灵活性
- ✅ **零风险**: 只是增加了一个字段别名，不改变业务逻辑
