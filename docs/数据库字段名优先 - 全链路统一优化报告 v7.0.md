# 数据库字段名优先 - 全链路统一优化报告 v7.0

## 📋 **核心原则**

> **"单一事实来源" (Single Source of Truth)**
> 
> 整个系统只认**数据库字段名**作为唯一标准，不需要任何转换！

### ✅ **三大统一**

1. **前端配置** → 使用数据库字段名（金蝶 K3 风格）
2. **后端实体** → 使用数据库字段名（金蝶 K3 风格）  
3. **数据库表** → 数据库字段名（金蝶 K3 风格）

### ❌ **零转换**

- ❌ 不需要 MyBatis 驼峰转换
- ❌ 不需要 `dbField` 映射（99% 场景）
- ❌ 不需要 `formBind` 配置
- ❌ 不需要字段名翻译

---

### **3. 前端配置精简**

**文件**: `销售订单初始化配置.sql`

#### **form_config 优化**

**修改前（冗余 dbField）** ❌：
```json
{
  "field": "FBillNo",
  "label": "单据编号",
  "dbField": "FBillNo"  // ❌ 冗余
}
```

**修改后（纯净）** ✅：
```json
{
  "field": "FBillNo",
  "label": "单据编号"
  // ✅ 无需 dbField
}
```

**特殊字段：销售员**（唯一保留 dbField 的场景）：
```json
{
  "field": "fseller",      // ✅ 前端绑定（小写，兼容字典返回）
  "dbField": "FSalerId"    // ✅ 查询条件（数据库字段）
}
```

**原因**：
- 字典返回的数据是 `{ fseller: "S001" }`（小写）
- 数据库字段是 `FSalerId`（大写 F）
- 需要 `dbField` 进行映射

#### **table_config 优化**

移除所有表格列的 `dbField`，直接使用 `prop`：
```json
{"prop": "FBillNo", "label": "单据编号"}
```

#### **detail_config 优化**

移除所有字段的 `dbField`，直接使用 `prop`。

## ⚠️ **特殊情况说明**

### **销售员字段映射**

**为什么需要特殊处理？**

1. **历史原因**：
   - 字典接口返回 `fseller`（小写）
   - 数据库字段是 `FSalerId`（大写 F）

2. **兼容性考虑**：
   - 前端已经习惯使用 `fseller`
   - 字典构建器返回的数据结构固定

**解决方案**：

#### **方案 A：前端配置保留 dbField（当前方案）** ✅
```json
{
  "field": "fseller",      // 前端绑定
  "dbField": "FSalerId"    // 查询映射
}
```

#### **方案 B：后端增加适配层**
```java
// 在 Service 层统一处理
if (StringUtils.isNotBlank(order.getFseller())) {
    order.setFSalerId(order.getFseller());
}
```

#### **方案 C：修改字典返回（推荐未来实施）** ⭐
```javascript
// 修改字典构建器，返回数据库字段名
{
  value: "user123",
  label: "张三",
  FSalerId: "S001"  // ✅ 改为数据库字段名
}
```

**优点**：彻底统一，无需特殊处理

**缺点**：需要修改字典构建器逻辑
