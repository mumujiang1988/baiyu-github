# 销售员字段映射异常消除方案 v8.0

## 📋 **问题回顾**

### **历史遗留问题**

销售员字段是整个系统中**唯一需要保留 `dbField` 映射**的异常场景。

#### **修改前的三方不一致** ❌

| 位置 | 字段名 | 类型 | 说明 |
|------|--------|------|------|
| **数据库表** `t_sale_order` | `FSalerId` | varchar(50) | ✅ 数据库字段（大写 F） |
| **员工表** `sys_employee` | `salesman_id` | varchar | ✅ 员工销售员编码（小写） |
| **字典返回** | `fseller` | varchar | ⚠️ 别名（小写，来自 salesman_id） |
| **前端配置** | `field: "fseller"` | - | ❌ 被迫使用小写 |
| **前端配置** | `dbField: "FSalerId"` | - | ❌ 需要额外映射 |

**数据流**：
```
sys_employee.salesman_id (小写)
    ↓ AS fseller
字典返回 { fseller: "S001" }
    ↓ 
前端绑定 formData.fseller
    ↓ 
查询条件 WHERE FSalerId = ? (需要 dbField 映射)
```

---

## 🎯 **根本原因分析**

### **为什么会出现字段名不一致？**

#### **1. 历史习惯**
- 前端开发习惯使用小写字段名（如 `fseller`, `fBillNo`）
- 认为这样更符合 JavaScript 命名规范

#### **2. 误解 MyBatis 映射**
- 认为必须通过 `AS` 别名将数据库字段转为小写
- 担心直接使用数据库字段名会导致前端不兼容

#### **3. 缺乏统一标准**
- 没有强制要求"数据库字段名优先"
- 各模块自行其是，造成混乱

---

## ✅ **v8.0 完整解决方案**

### **核心原则：彻底统一**

> **"让字典返回适配数据库字段名，而不是反过来！"**

---

### **修改 1：后端字典返回字段名** 🔧

**文件**: `ErpDictionaryMapper.java`

#### **修改前** ❌
```java
@Select("SELECT " +
        "  e.salesman_id AS fseller, " +  // ❌ 小写别名
        "  ...")
List<Map<String, Object>> selectSalespersonsDict();
```

**返回数据**：
```javascript
{
  value: "user123",
  label: "张三 - 销售部 - S001",
  fseller: "S001"  // ❌ 小写
}
```

#### **修改后** ✅
```java
@Select("SELECT " +
        "  e.salesman_id AS FSalerId, " +  // ✅ 使用数据库字段名
        "  ...")
List<Map<String, Object>> selectSalespersonsDict();
```

**返回数据**：
```javascript
{
  value: "user123",
  label: "张三 - 销售部 - S001",
  FSalerId: "S001"  // ✅ 与数据库字段名一致
}
```

---

### **修改 2：前端配置移除 dbField** 🧹

**文件**: `销售订单初始化配置.sql`

#### **修改前** ❌
```json
{
  "field": "fseller",           // ❌ 小写
  "label": "销售员",
  "dictionary": "salespersons",
  "dbField": "FSalerId"         // ❌ 需要额外映射
}
```

**问题**：
- 需要维护 `field` 和 `dbField` 两个字段
- 增加了配置复杂度
- 容易造成混淆

#### **修改后** ✅
```json
{
  "field": "FSalerId",          // ✅ 直接使用数据库字段名
  "label": "销售员",
  "dictionary": "salespersons"
  // ✅ 无需 dbField
}
```

**优势**：
- ✅ 配置更简洁
- ✅ 无需维护映射关系
- ✅ 全链路统一

---

### **修改 3：前端表单绑定自动适配** 🔄

**文件**: `BusinessConfigurable.vue`

#### **数据流（修改后）**

```javascript
// 字典返回
{
  value: "user123",
  FSalerId: "S001"  // ✅ 直接匹配
}

// 表单绑定
v-model="formData[FSalerId]"  // ✅ 自动绑定到 FSalerId

// 提交数据
submitData: {
  FSalerId: "S001"  // ✅ 直接提交
}

// SQL 查询
WHERE FSalerId = 'S001'  // ✅ 无需转换
```

**全程无转换！** 🎉

## ⚠️ **注意事项**

### **1. 兼容性考虑**

**影响范围**：
- ✅ 所有使用销售人员字典的页面
- ✅ 所有依赖 `fseller` 字段的代码

**需要检查的文件**：
```bash
# 查找所有使用 fseller 的地方
grep -r "fseller" baiyu-web/src/views/erp/

# 查找所有使用 FSalerId 的地方
grep -r "FSalerId" baiyu-web/src/views/erp/
```

---

### **2. 可能的副作用**

#### **副作用 1：旧代码引用 `fseller`**

**现象**：
```javascript
// ❌ 旧代码可能直接使用 fseller
const salerCode = response.data.fseller;
```

**修复**：
```javascript
// ✅ 改为使用 FSalerId
const salerCode = response.data.FSalerId;
```

---

#### **副作用 2：PowerShell 脚本推广错误字段**

**现象**：
```powershell
# ❌ fix-salesperson-query.ps1 在推广使用 fseller
$content = $content -replace 'v-model="formData\.FSalerId"', 'v-model="formData.fseller"'
```

**修复**：
- 停止执行该脚本
- 或者修改脚本逻辑（反向替换）

---

### **3. 数据迁移（如有必要）**

如果数据库中已经存在使用 `fseller` 字段名的数据（可能性极低），需要：

```sql
-- 检查是否有异常数据
SELECT * FROM t_sale_order 
WHERE FSalerId IS NULL OR FSalerId = '';

-- 如有必要，进行数据修复
UPDATE t_sale_order 
SET FSalerId = ? 
WHERE ...;
```
