## 📊 **问题分析**

### **1. 销售人员字典查询逻辑**

#### **后端接口**: `/erp/engine/dict/union/salespersons`

**SQL 查询** (ErpDictionaryMapper.java):
```sql
SELECT 
  CONCAT(u.nick_name, IFNULL(CONCAT('(', d.dept_name, ')'), ''), 
         IFNULL(CONCAT(' - ', e.salesman_id), '')) AS label,
  u.user_id AS value,                    -- ⚠️ 返回的是 user_id
  'salespersons' AS type,
  u.nick_name AS nickName,
  d.dept_name AS departmentName,
  e.salesman_id AS fseller,              -- ✅ salesman_id 在 fseller 字段
  GROUP_CONCAT(DISTINCT sr.role_name ORDER BY sr.role_id) AS roleNames
FROM sys_user u
LEFT JOIN sys_dept d ON u.dept_id = d.dept_id
LEFT JOIN sys_user_role ur ON u.user_id = ur.user_id
LEFT JOIN sys_role sr ON ur.role_id = sr.role_id
LEFT JOIN sys_employee e ON u.staff_id = e.fid
WHERE (
  u.dept_id IN ('1995775271620800513', '1995776039019048962', 
                '1995776549579091969', '1995776618810273794')
  OR ur.role_id IN (1, 2016378335548186625, 2021458108021686273)
)
AND u.status = '1'
AND u.del_flag = '0'
GROUP BY u.user_id, u.nick_name, u.staff_id, d.dept_name, e.salesman_id
ORDER BY u.nick_name
```

#### **返回数据结构**
```json
[
  {
    "label": "张三 (销售部 - 100949)",
    "value": 1234567890,           // ← user_id (雪花算法 ID)
    "type": "salespersons",
    "nickName": "张三",
    "departmentName": "销售部",
    "fseller": "100949"            // ← salesman_id (销售员编码)
  }
]
```

### **2. 核心问题**

#### **问题 1: 字段映射不一致**
- ❌ **前端使用**: `FSalerId` (金蝶 K3 字段命名风格)
- ✅ **后端返回**: `fseller` (小写 + 下划线)
- ⚠️ **值类型不匹配**: 
  - 前端期望：销售员编码字符串 (如 "100949")
  - 后端返回：`user_id` (雪花算法 ID，如 1234567890)

#### **问题 2: 查询条件传递错误**
```
用户输入：100949 (销售员编码)
   ↓
前端处理：可能进行了某种转换或映射
   ↓
后端接收：100055 (完全不同的值)
```

**可能的原因**:
1. 前端表单组件使用了错误的字段绑定
2. 数据转换逻辑有误
3. 缓存了旧的销售员数据
4. 多个销售员编码映射混乱

@Select("<script>" +
        "SELECT " +

## 🎯 **最佳实践建议**

### **1. 字段命名规范**

```javascript
// ✅ 推荐：前后端统一的命名
{
  fseller: "100949",        // 销售员编码
  fseller_name: "张三",     // 销售员姓名
  fseller_dept: "销售部"    // 销售员部门
}

// ❌ 不推荐：混用多种命名风格
{
  FSalerId: "100949",       // 金蝶风格
  salerName: "张三",        // 驼峰风格
  seller_dept: "销售部"     // 下划线风格
}
```

### **2. 字典值设计原则**

- **唯一性**: 确保 value 字段能唯一标识一个选项
- **稳定性**: 使用业务编码而非自增 ID
- **可读性**: 便于调试和日志分析
- **一致性**: 与数据库字段保持一致

- 在后端添加详细日志，记录查询条件的来源
