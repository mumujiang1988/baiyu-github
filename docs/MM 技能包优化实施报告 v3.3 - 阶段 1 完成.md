#### 6. N+1 查询预防（性能关键）

**内容概览**:
- ✅ **典型反例剖析**
  - for 循环中的数据库查询
  - 100 个订单 = 101 次 SQL 查询

- ✅ **三种解决方案**
  1. MyBatis-Plus `<select>` 联查（推荐）
     - 主查询 + 批量查询 = 2 次 SQL
  2. MyBatis `<association>` 嵌套查询（XML 方式）
     - resultMap 自动关联
  3. 手动 JOIN（最灵活）
     - @Select 注解或 XML

- ✅ **检测工具**
  - MyBatis SQL 日志开启
  - 识别 N+1 模式

**代码量**: ~120 行  
**适用场景**: 关联查询、列表查询

---

#### 7. 连接池配置（生产必备）

**内容概览**:
- ✅ **HikariCP 完整配置**
  - minimum-idle: 10
  - maximum-pool-size: 20
  - connection-timeout: 30000
  - idle-timeout: 600000
  - max-lifetime: 1800000

- ✅ **配置计算公式**
  - CPU 密集型：CPU 核心数 + 1
  - IO 密集型：CPU 核心数 × 2 + spindle 数量
  - 快速估算：CPU 核心数 × 2 + 1

- ✅ **监控连接池**
  - HikariPoolMXBean 使用
  - 活跃连接数、空闲连接数监控

- ✅ **常见问题排查**
  - 连接泄漏：try-with-resources
  - 连接耗尽：增大 pool size + 优化慢查询

**代码量**: ~80 行  
**适用场景**: 生产环境部署

1. **渐进式改进优于颠覆式重构**

   - 避免大规模改动带来的风险

2. **技术栈适配至关重要**
   - 不盲目照搬其他框架的最佳实践
   - 只吸收与现有技术栈兼容的内容

3. **方法论比具体内容更有价值**
   - skill-creator 的评估循环方法论 > 具体测试用例
   - fullstack-dev 的性能优化理念 > 具体实现代码
