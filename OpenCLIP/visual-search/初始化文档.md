## 详细初始化步骤

### 一、MySQL 数据库初始化

#### 1.1 自动初始化流程

当您执行 `docker-compose up -d` 时，MySQL 会自动：

1. 创建数据库 `visual_search`
2. 创建用户 `vs_user`（密码：`vs_pass123`）
3. 按字母顺序执行 `/docker-entrypoint-initdb.d/` 目录下的 SQL 文件

**SQL 文件执行顺序：**

```
01_init.sql          ← 首先执行（创建表结构）
02_fix_unique_index.sql  ← 然后执行（修复索引）
```

**创建的表结构：**

| 表名 | 说明 | 关键字段 |
|------|------|----------|
| `product` | 产品信息表 | id, product_code, name, spec, category, status |
| `product_image` | 产品图片表 | id, product_code, image_path, image_hash, milvus_id |
| `search_log` | 检索日志表 | id, query_image_hash, top_product_code, similarity_score |
| `ingest_log` | 入库日志表 | id, product_code, image_count, success_count, fail_count |

#### 1.2 验证 MySQL 初始化

```powershell
# 检查容器状态
docker-compose ps mysql

# 查看表是否创建成功
docker exec visual-search-mysql mysql -uvs_user -pvs_pass123 visual_search -e "SHOW TABLES;"

# 预期输出：
# Tables_in_visual_search
# ingest_log
# product
# product_image
# search_log

# 查看表结构
docker exec visual-search-mysql mysql -uvs_user -pvs_pass123 visual_search -e "DESCRIBE product;"
```

#### 1.3 重新初始化 MySQL

如果表未创建或需要重置数据：

```powershell
# ⚠️ 警告：这将删除所有数据！

# 1. 停止并删除所有容器和数据卷
docker-compose down -v

# 2. 重新启动（会重新执行初始化脚本）
docker-compose up -d

# 3. 等待 15 秒让 MySQL 完成初始化
Start-Sleep -Seconds 15

# 4. 验证表已创建
docker exec visual-search-mysql mysql -uvs_user -pvs_pass123 visual_search -e "SHOW TABLES;"
```

#### 1.4 常见 MySQL 问题

**问题 1：表不存在**

原因：SQL 文件执行顺序错误

解决：确保文件名使用数字前缀
```
✅ 01_init.sql
✅ 02_fix_unique_index.sql

❌ init.sql（会在 fix 之后执行）
❌ fix_unique_index.sql
```

**问题 2：权限被拒绝**

原因：数据卷权限问题

解决：
```powershell
# 删除数据卷
docker volume rm visual-search_mysql_data

# 重新启动
docker-compose up -d
```

### 二、Milvus 向量数据库初始化

#### 2.1 依赖关系

Milvus 依赖于以下服务，必须按顺序启动：

```
Etcd → MinIO → Milvus → Backend
```

Docker Compose 会自动处理依赖关系（通过 `depends_on`）。

#### 2.2 后端自动初始化 Milvus

当后端服务启动时，会自动执行以下操作：

**代码位置：** `backend/main.py` - startup 事件

```python
# 自动执行的步骤：
1. 连接 Milvus 服务器 (milvus:19530)
2. 创建集合 "product_images"
   - 字段定义：
     * id: INT64 (主键，自增)
     * product_code: VARCHAR(50)
     * image_id: INT64
     * embedding: FLOAT_VECTOR (维度 512)
3. 创建向量索引
   - 索引类型: IVF_FLAT
   - 度量方式: L2 (欧氏距离)
   - nlist: 128
4. 加载集合到内存（用于搜索）
```

#### 2.3 验证 Milvus 初始化

```powershell
# 1. 检查 Milvus 容器状态
docker-compose ps milvus

# 2. 查看后端日志确认 Milvus 连接
docker logs visual-search-backend 2>&1 | Select-String "Milvus"

# 预期输出：
# ✅ Milvus 连接成功
# 📝 创建集合: product_images
# 🔍 创建向量索引...
# ✅ 集合加载成功

# 3. 检查 Milvus 日志
docker logs visual-search-milvus --tail 30

# 应该看到正常的运行日志，没有 ERROR
```

#### 2.4 重新初始化 Milvus

如果 Milvus 出现问题：

```powershell
# 1. 重启 Milvus 及其依赖
docker-compose restart etcd minio milvus

# 2. 等待 Milvus 就绪
Start-Sleep -Seconds 30

# 3. 重启后端触发重新初始化
docker-compose restart backend

# 4. 验证连接
Start-Sleep -Seconds 10
docker logs visual-search-backend --tail 20 | Select-String "Milvus"
```

#### 2.5 常见 Milvus 问题

**问题 1：连接超时**

原因：etcd 或 MinIO 未就绪

解决：
```powershell
# 检查依赖服务
docker-compose ps etcd minio

# 如果未运行，重启所有服务
docker-compose restart etcd minio milvus
```

**问题 2：集合已存在**

这是正常现象，后端会跳过创建。如需重建：

```powershell
# 需要手动删除集合（通过 Python 脚本）
# 或删除整个 Milvus 数据卷
docker volume rm visual-search_milvus_data
docker-compose up -d milvus
```

后端服务启动时会自动创建用于存储产品图片的 bucket：

**代码位置：** `backend/services/storage_service.py`

```python
# 自动创建的 bucket: product-images
# 如果 bucket 已存在，会跳过创建
```

- 用户名: `minioadmin`
- 密码: `minioadmin`

```powershell
docker logs visual-search-backend 2>&1 | Select-String "MinIO"
```

## 验证与测试

### 完整验证脚本

创建文件 `verify-init.ps1`：

```powershell
# verify-init.ps1 - 系统初始化验证脚本

Write-Host "========================================" -ForegroundColor Cyan
Write-Host "  OpenCLIP 以图搜品系统 - 初始化验证" -ForegroundColor Cyan
Write-Host "========================================" -ForegroundColor Cyan
Write-Host ""

$ErrorCount = 0

# 1. 检查所有服务状态
Write-Host "[1/6] 检查服务状态..." -ForegroundColor Yellow
$services = docker-compose ps --format json | ConvertFrom-Json
foreach ($service in $services) {
    if ($service.State -eq "running") {
        Write-Host "  ✅ $($service.Name): Running" -ForegroundColor Green
    } else {
        Write-Host "  ❌ $($service.Name): $($service.State)" -ForegroundColor Red
        $ErrorCount++
    }
}
Write-Host ""

# 2. 检查 MySQL 表
Write-Host "[2/6] 检查 MySQL 表结构..." -ForegroundColor Yellow
try {
    $tables = docker exec visual-search-mysql mysql -uvs_user -pvs_pass123 visual_search -e "SHOW TABLES;" 2>$null
    if ($tables -match "product" -and $tables -match "product_image") {
        Write-Host "  ✅ MySQL 表创建成功 (4 张表)" -ForegroundColor Green
    } else {
        Write-Host "  ❌ MySQL 表缺失" -ForegroundColor Red
        $ErrorCount++
    }
} catch {
    Write-Host "  ❌ MySQL 连接失败: $_" -ForegroundColor Red
    $ErrorCount++
}
Write-Host ""

# 3. 检查 Milvus 连接
Write-Host "[3/6] 检查 Milvus 向量数据库..." -ForegroundColor Yellow
$milvusLog = docker logs visual-search-backend 2>&1 | Select-String "Milvus 连接成功"
if ($milvusLog) {
    Write-Host "  ✅ Milvus 连接成功" -ForegroundColor Green
} else {
    Write-Host "  ❌ Milvus 连接失败" -ForegroundColor Red
    $ErrorCount++
}
Write-Host ""

# 4. 检查 MinIO 健康状态
Write-Host "[4/6] 检查 MinIO 对象存储..." -ForegroundColor Yellow
$minioHealth = docker inspect visual-search-minio --format='{{.State.Health.Status}}' 2>$null
if ($minioHealth -eq "healthy") {
    Write-Host "  ✅ MinIO 健康状态: healthy" -ForegroundColor Green
} else {
    Write-Host "  ⚠️  MinIO 健康状态: $minioHealth" -ForegroundColor Yellow
}
Write-Host ""

# 5. 检查后端 API
Write-Host "[5/6] 检查后端 API..." -ForegroundColor Yellow
try {
    $response = Invoke-WebRequest -Uri "http://localhost:8000/api/v1/products?page=1&page_size=1" -UseBasicParsing -TimeoutSec 5
    if ($response.StatusCode -eq 200) {
        Write-Host "  ✅ API 响应正常 (HTTP 200)" -ForegroundColor Green
    } else {
        Write-Host "  ⚠️  API 响应异常 (HTTP $($response.StatusCode))" -ForegroundColor Yellow
    }
} catch {
    Write-Host "  ❌ API 调用失败: $_" -ForegroundColor Red
    $ErrorCount++
}
Write-Host ""

# 6. 检查前端服务
Write-Host "[6/6] 检查前端服务..." -ForegroundColor Yellow
try {
    $response = Invoke-WebRequest -Uri "http://localhost:8080" -UseBasicParsing -TimeoutSec 5
    if ($response.StatusCode -eq 200) {
        Write-Host "  ✅ 前端服务正常 (HTTP 200)" -ForegroundColor Green
    } else {
        Write-Host "  ⚠️  前端响应异常 (HTTP $($response.StatusCode))" -ForegroundColor Yellow
    }
} catch {
    Write-Host "  ❌ 前端服务不可用: $_" -ForegroundColor Red
    $ErrorCount++
}
Write-Host ""

# 总结
Write-Host "========================================" -ForegroundColor Cyan
if ($ErrorCount -eq 0) {
    Write-Host "  ✅ 所有检查通过！系统初始化成功" -ForegroundColor Green
} else {
    Write-Host "  ⚠️  发现 $ErrorCount 个问题，请检查上述错误" -ForegroundColor Red
}
Write-Host "========================================" -ForegroundColor Cyan
Write-Host ""
Write-Host "访问地址：" -ForegroundColor Cyan
Write-Host "  前端界面: http://localhost:8080" -ForegroundColor White
Write-Host "  API 文档: http://localhost:8000/docs" -ForegroundColor White
Write-Host ""
```

## 常见问题

### Q1: 首次启动非常慢？

**原因：** 需要下载 OpenCLIP 模型（约 354MB）

**解决方案：**
- 耐心等待 2-3 分钟
- 模型会缓存到 Docker 卷中，后续启动很快
- 查看进度：`docker logs visual-search-backend -f`

### Q2: MySQL 表未创建？

**原因：** 数据卷已存在，不会重新执行初始化脚本

**解决方案：**
```powershell
docker-compose down -v
docker-compose up -d
Start-Sleep -Seconds 15
docker exec visual-search-mysql mysql -uvs_user -pvs_pass123 visual_search -e "SHOW TABLES;"
```

### Q3: Milvus 连接失败？

**原因：** etcd 或 MinIO 未就绪

**解决方案：**
```powershell
# 检查依赖服务
docker-compose ps etcd minio

# 重启相关服务
docker-compose restart etcd minio milvus
Start-Sleep -Seconds 30
docker-compose restart backend
```

### Q4: 前端显示 404 错误？

**原因：** 静态资源缺失

**解决方案：**
已修复！`public/vite.svg` 文件已创建。如果仍有问题：
```powershell
docker-compose up -d --build frontend
```

### Q5: 如何清空所有数据重新开始？

**解决方案：**
```powershell
# ⚠️ 警告：这将删除所有数据！
docker-compose down -v
docker-compose up -d
```

### Q6: 如何查看实时日志？

**解决方案：**
```powershell
# 查看所有服务日志
docker-compose logs -f

# 查看特定服务日志
docker-compose logs -f backend
docker-compose logs -f mysql
docker-compose logs -f milvus
```

### Q7: 如何备份数据？

**解决方案：**
```powershell
# 备份 MySQL 数据
docker exec visual-search-mysql mysqldump -uvs_user -pvs_pass123 visual_search > backup_$(Get-Date -Format "yyyyMMdd").sql

# 备份 Docker 卷
docker run --rm -v visual-search_mysql_data:/data -v ${PWD}:/backup alpine tar czf /backup/mysql_backup.tar.gz -C /data .
```

### Q8: 如何更新代码后重新部署？

**解决方案：**
```powershell
# 重新构建并启动
docker-compose up -d --build

# 或者只重建特定服务
docker-compose up -d --build backend
docker-compose up -d --build frontend
```

## 性能优化建议

### 1. 内存分配

编辑 Docker Desktop 设置：
- CPU: 4 核
- 内存: 8GB（最低），16GB（推荐）
- Swap: 2GB

### 2. Milvus 性能调优

对于大规模数据（>100万向量）：

```yaml
# docker-compose.yml 中添加环境变量
milvus:
  environment:
    - KNOWHERE_GPU_MEM_POOL_SIZE=2048;4096
    - PROXY_MAX_CONNECTION_NUM=1024
```

### 3. MySQL 优化

```sql
-- 在 MySQL 中执行
SET GLOBAL innodb_buffer_pool_size = 2147483648;  -- 2GB
SET GLOBAL innodb_log_file_size = 512M;
```

### 4. 模型缓存

模型文件存储在 Docker 卷中，确保有足够空间：
```powershell
# 检查卷大小
docker system df -v | Select-String "backend_models"
```

## 安全注意事项

### 1. 修改默认密码

生产环境务必修改默认密码：

```yaml
# docker-compose.yml
mysql:
  environment:
    MYSQL_ROOT_PASSWORD: your_strong_password
    MYSQL_PASSWORD: your_strong_password

minio:
  environment:
    MINIO_ACCESS_KEY: your_access_key
    MINIO_SECRET_KEY: your_secret_key
```

### 2. 限制端口暴露

仅暴露必要的端口：

```yaml
# 开发环境
ports:
  - "8000:8000"
  - "8080:80"

# 生产环境（使用反向代理）
# 不直接暴露数据库端口
```

### 3. 启用 HTTPS

使用 Nginx 反向代理并配置 SSL 证书。

### B. 环境变量说明

| 变量名 | 默认值 | 说明 |
|--------|--------|------|
| MYSQL_HOST | mysql | MySQL 主机 |
| MYSQL_PORT | 3306 | MySQL 端口 |
| MYSQL_USER | vs_user | MySQL 用户名 |
| MYSQL_PASSWORD | vs_pass123 | MySQL 密码 |
| MYSQL_DB | visual_search | 数据库名 |
| MILVUS_HOST | milvus | Milvus 主机 |
| MILVUS_PORT | 19530 | Milvus 端口 |
| OPENCLIP_MODEL | ViT-B-32 | OpenCLIP 模型名称 |
| OPENCLIP_PRETRAINED | openai | 预训练权重来源 |

### C. 端口映射总览

## 技术支持

如遇到问题，请提供以下信息：

1. **系统信息**
   ```powershell
   docker --version
   docker-compose --version
   ```

2. **服务状态**
   ```powershell
   docker-compose ps
   ```

3. **错误日志**
   ```powershell
   docker-compose logs --tail=100 > logs.txt
   ```

4. **复现步骤**
   - 详细描述操作步骤
   - 预期结果 vs 实际结果
